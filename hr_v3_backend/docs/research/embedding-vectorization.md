# 数据向量化（Embedding）调查报告

> HR 简历筛选系统场景 | 2026-02

---

## 一、什么是数据向量化

**数据向量化**（Embedding）是将文本转换为高维数值向量的过程。语义相近的内容在向量空间中距离更近。

```
"前端开发工程师，5年React经验" -> [0.023, -0.156, 0.891, ..., 0.034]  // 1536维
"Frontend developer, 5 years React" -> [0.025, -0.148, 0.887, ..., 0.031]  // 极为相似
```

### 与传统关键词搜索的区别

| 维度 | 关键词搜索（BM25） | 向量语义搜索（Embedding） |
|------|------|------|
| 匹配方式 | 精确词频匹配 | 语义含义匹配 |
| "React"能否匹配"前端框架" | 不能 | 能 |
| 同义词 | 需手动配置同义词表 | 自动理解"开发者"="程序员" |
| 跨语言 | 无 | 多语言模型可跨语言匹配 |
| 精确术语匹配 | 强（如"CPA"、"PMP"） | 弱（可能泛化过度） |
| 可解释性 | 高（能看到匹配了哪些词） | 低（向量是黑盒） |

## 二、在 HR 简历筛选场景中的作用

### 解决的问题

1. **词汇不匹配**：简历写"Django"但不写"Python"，关键词搜索无法建立关联
2. **同义词盲区**："软件工程师"和"开发者"是同一角色，关键词无法识别
3. **上下文不敏感**："带领5人团队"暗示领导力，关键词无法捕捉
4. **跨语言简历**：中英混写简历难以统一处理
5. **规模化瓶颈**：当前 LLM 逐份评分，500份需500次 API 调用

### 应用场景

| 场景 | 说明 | 效果 |
|------|------|------|
| 简历-JD 快速预筛 | JD 向量 vs 所有简历向量，毫秒级返回 Top-N | 替代逐份 LLM 评分的前置过滤 |
| 相似候选人发现 | 输入一份优秀简历，找到背景最相似的人 | HR 找备选人选 |
| 技能语义理解 | "Vue.js"和"React"都被理解为"前端框架" | 减少因措辞不同的漏筛 |
| 跨职位推荐 | 后端候选人可能也适合全栈职位 | 自动发现跨职位匹配 |

### 与当前 MiniMax LLM 评分的关系：互补

| | 向量化（Embedding） | LLM 评分（MiniMax） |
|------|------|------|
| 角色 | 第一轮快速筛选 | 第二轮深度评估 |
| 速度 | 毫秒级，可批量 | 秒级，逐份处理 |
| 成本 | 极低（~$0.01/千份） | 较高（~$0.50/50份） |
| 输出 | 相似度分数（0~1） | 结构化评分 + 文字解释 |

**推荐两级架构：**

```
500份简历
    │
    ▼
[第一级] 向量相似度预筛 → Top 50（毫秒级，~$0.01）
    │
    ▼
[第二级] MiniMax LLM 深度评分 → 排序+评级（分钟级，~$0.50）
    │
    ▼
HR 查看 Top 50 详细评分
```

成本节约：减少 90% 的 LLM API 调用。

## 三、优缺点

### 优点

1. **语义理解强**：自动理解"全栈"≈"前端+后端"，无需同义词表
2. **多语言支持**：中英混写简历可统一处理
3. **检索极快**：百万级数据 <10ms
4. **成本极低**：向量化1000份简历约 $0.02
5. **可扩展**：新增简历只需一次 Embedding，后续查询自动包含

### 缺点

1. **可解释性差**：无法解释为什么两份简历相似度是 0.85
2. **精确匹配弱**：特定证书（CPA、PMP）不如关键词可靠
3. **模型依赖**：换模型需全量重建向量
4. **维护复杂度**：增加 pgvector 扩展和 Embedding 服务
5. **领域适配**：通用模型可能不懂"J2EE已过时，Spring Boot是后继"

### 什么时候该用 vs 不该用

| 该用 | 不该用 |
|------|------|
| 候选人池 >100 人 | 候选人池 <20 人 |
| 简历中英混写 | 只需精确匹配证书资质 |
| 需要"找相似的人"功能 | 团队不愿增加系统复杂度 |
| 想降低 LLM API 成本 | 数据量极小，投入产出不成比例 |

## 四、主流技术栈

### 4.1 Embedding 模型

| 模型 | 维度 | 中文支持 | 许可证 | 价格 | 推荐场景 |
|------|------|------|------|------|------|
| **OpenAI text-embedding-3-small** | 1536 | 良好 | 商业 API | $0.02/M tokens | **首选：低成本、易集成** |
| **BGE-M3 (BAAI)** | 1024 | 最优之一 | MIT | 免费自托管 | 中文最优、开源商用 |
| **Qwen3-Embedding** | 32-4096可调 | 优秀 | Apache 2.0 | 免费自托管 | MTEB多语言 #1 |
| OpenAI text-embedding-3-large | 3072 | 良好 | 商业 API | $0.13/M tokens | 高精度需求 |
| Cohere Embed v4 | 1536 | 支持 | 商业 API | $0.12/M tokens | 企业级、多模态 |
| Jina Embeddings v3 | 可调 | 89语言 | CC-BY-NC | API付费 | 多语言长文档 |
| MiniMax embo-01 | — | 中文 | 商业 API | 待确认 | 保持单一供应商 |

**当前项目推荐：OpenAI text-embedding-3-small**
- 已有 `@ai-sdk/openai` 依赖，集成最简
- 成本最低，1000份简历 $0.02
- 中文效果足够 HR 场景使用

### 4.2 向量数据库

| 数据库 | 类型 | 与当前系统兼容性 | 优势 | 劣势 |
|------|------|------|------|------|
| **pgvector** | PG 扩展 | **最高（已有 PG）** | 零新基础设施，Drizzle 原生支持 | 超大规模性能不如专用 DB |
| Qdrant | 专用向量 DB | 低（需新服务） | 性能极高 | 运维复杂度增加 |
| Pinecone | 托管服务 | 低 | 免运维 | $70+/月起 |
| Chroma | 嵌入式 | 中 | 轻量 | 仅适合小规模 |

**当前项目最佳选择：pgvector**
- 已用 PostgreSQL + Drizzle ORM，只需 `CREATE EXTENSION vector;`
- Drizzle 原生支持 `vector` 类型、`cosineDistance`、HNSW 索引
- HR 场景数据量（<100万），pgvector 完全胜任
- 0.8.0 版本性能比 0.7.4 快 5.7 倍

### 4.3 混合搜索（Hybrid Search）

2025-2026 年生产级 RAG 的标准做法，结合两种搜索的优势：

```
查询："有5年React经验，必须有CPA证书"
    │
    ├─ [向量搜索] 语义理解"React经验" → 找到写"前端开发"的候选人
    │
    ├─ [BM25关键词] 精确匹配"CPA" → 确保不遗漏特定资质
    │
    ▼
[RRF 融合] 合并两路结果 → 最终排序
```

PostgreSQL 原生支持：`tsvector`（关键词）+ pgvector（语义）可在同一 SQL 中组合。

## 五、与当前系统的集成方案

### 技术栈兼容性

| 组件 | 当前状态 | 改造难度 |
|------|------|------|
| PostgreSQL | 已有 | 只需安装 pgvector 扩展 |
| Drizzle ORM | v0.45+ | 原生支持 vector 类型 |
| `@ai-sdk/openai` | 已安装 | 可复用调 Embedding API |
| Bun 运行时 | 已有 | pgvector-node 支持 Bun |

### MVP 实现概要

**1. 安装 pgvector**
```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

**2. Schema 添加 embedding 列**
```typescript
// resumes 表添加
embedding: vector("embedding", { dimensions: 1536 }),
// 加 HNSW 索引
index("resumes_embedding_idx").using("hnsw", table.embedding.op("vector_cosine_ops"))
```

**3. Embedding 服务**
```typescript
// 调用 OpenAI text-embedding-3-small
async function generateEmbedding(text: string): Promise<number[]> { ... }
```

**4. 向量搜索**
```typescript
// Drizzle 原生语法
const similarity = sql`1 - (${cosineDistance(resumes.embedding, queryEmbedding)})`;
const results = await db.select({ resumeId, candidateId, similarity })
  .from(resumes)
  .where(gt(similarity, 0.5))
  .orderBy(desc(similarity))
  .limit(20);
```

## 六、2025-2026 趋势

1. **Qwen3-Embedding 登顶 MTEB 多语言榜**（2025.06），支持指令感知和灵活维度
2. **小模型逆袭**：118M 参数的 e5-small 在 Top-5 准确率达 100%，超越所有 7B 大模型
3. **向量能力回归关系型数据库**：pgvector 成为主流，专用向量数据库份额下降
4. **混合搜索成标准**：BM25 + 向量 + Reranker 三级管道是生产标配
5. **Matryoshka 表示学习**：支持动态截断维度（1536→256），损失 <1%

## 七、结论

向量化对当前系统是**高收益、低成本、低风险**的增强：

- **与 MiniMax LLM 互补**：向量做快速粗筛，LLM 做深度评分
- **技术栈完美兼容**：pgvector + Drizzle 原生支持，零新基础设施
- **成本可忽略**：月度 ~$0.02
- **实施周期短**：MVP ~5天
- **行业验证**：Resume2Vec 等研究证明向量化匹配比传统 ATS 提升 15%+ 精度

**建议：在当前系统已稳定的基础上，作为 P2 功能规划实施。**
